## Challenges & Systemic Issues

### 1. Workflow / tracking drift (Story 3.3)

- Guide was authored and sprint marked **done** before `dev-story` closed the story file.
- Story file still shows `ready-for-dev` tasks unchecked while deliverable exists on disk.
- **Root cause:** Bulk implementation skipped BMad status transitions (create-story → dev-story → code-review → done).

### 2. Scope boundaries blurred across Epic 3 stories

- Story 3.3 spec excluded Git (3.5) and full `launch.json` (3.4); guide includes both.
- **Impact:** Acceptable for a single facilitator document, but story-level AC tracking becomes ambiguous.
- **Decision pending:** Keep merged guide vs. trim to cross-references.

### 3. Pedagogy vs. speed tradeoff (AC #3)

- Step 1 uses PowerShell here-string bulk CSV creation without teaching column semantics.
- Code review flagged weak manual-data literacy vs. copy-paste speed.

### 4. Sprint-status inconsistency

- `epic-3: done` while `3-3-author-lab-03-search-app-guide-md: in-progress`.
- Epic completion should not be marked until all stories reach `done`.

### 5. Missing story files

- 3.1, 3.2, 3.5 lack artifact story files — lessons and review history not captured per story.

## Key Learnings

1. **Documentation stories need the same workflow closure as code stories** — unchecked tasks and stale status erode trust in sprint tracking.
2. **Lab guides should use portable paths** — `$PSScriptRoot` or relative `cd` from repo root, never author machine paths.
3. **Merged facilitator docs benefit from explicit scope notes** — when 3.3/3.4/3.5 content lives in one file, add a short "ownership map" section.
4. **Review before marking done** — AC #3 (pedagogy) and verification matrix completeness are easy to miss in large markdown deliverables.
5. **Epic 1–2 retros still optional** — no prior retro action items to follow through on; this is the first completed retrospective in the project.
