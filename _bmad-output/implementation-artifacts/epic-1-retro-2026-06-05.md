## What Went Well

**Project Lead assessment:** Overall delivery was successful — no major concerns raised.

### Success Themes

1. **Frozen API contract preserved** — Frontend unchanged; search semantics identical to CSV baseline via anchored case-insensitive `$regex` with escaped user input.
2. **Clean module boundaries** — `lib/db.js` (connection), `lib/seed.js` (CSV upsert), `models/User.js` (schema), `server.js` (orchestration). Story 1.4 was a surgical stub replacement.
3. **Architectural restraint held (SM-C1)** — No repository/service layers, no `lib/search.js`, no over-engineering.
4. **Lift-and-shift seed pattern** — Moving `loadUsers()` validation from `server.js` into `lib/seed.js` with `bulkWrite` upsert on `email` worked cleanly.
5. **Integration test contract closed** — Story 1.5 added graceful skip (exit 0) when `MONGODB_URI` unset; explicit URI passthrough in spawn env.
6. **Service README aligned** — MongoDB workflow documented with troubleshooting table; test matrix counts preserved.

## Challenges & Growth Areas

### Common Struggles

| Pattern | Stories Affected | Impact |
|---------|------------------|--------|
| Atlas `.env` dependency for full verification | 1.3, 1.4, 1.5 | Fail-fast paths tested locally; happy-path Atlas integration often marked "reviewer action" |
| Node engine mismatch (`mongoose@9.x` requires Node ≥20.19; `package.json` declares ≥18) | 1.1, 1.2, 1.3 | Documented but not resolved — cohort risk before next lab |
| Intentional regression windows between stories | 1.3 → 1.5 | `npm test` broken mid-epic until Story 1.5; expected but fragile |
| Scope bleed | 1.3 | MongoDB search shipped early; reconciled in Story 1.4 review |

### Recurring Deferred Review Items

- CORS unrestricted (workshop scope)
- No MongoDB readiness signal in `/health`
- No auth or rate limiting on `/api/search`
- No graceful shutdown (`SIGTERM` / `mongoose.disconnect`)
- `verify-lab.ps1` PS 5.1 empty error body on HTTP 400 (no-params check)
- Shared Atlas cluster pollution risk for strict count assertions in tests

### Key Insights

1. **Story boundaries with external dependencies** — When Atlas is required, document which ACs need facilitator `.env` vs. which can be verified without it.
2. **Regression windows are acceptable** — But communicate them clearly so mid-epic test failures aren't treated as regressions.
3. **Documentation split is intentional** — Service README (Epic 1) vs. lab guide + scripts (Epic 2) — facilitators still need Epic 2 before next cohort.
